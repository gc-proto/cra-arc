Correct use

  • Before including any entry fields for the form, provide a one line statement about what this form is used for in the following format: "Use this form to ...". This assures the user that the form they are filling out is the correct form
  • When the form is spread over multiple screens/pages, provide a progress bar to let the user know at what stage in the completion process they are at

Steps form

Use to provide the ability to split a form into steps.

How to implement

  1. Create a <div> element with class wb-steps (you can also add class wb-frmvld for validation). 
    <div class="wb-steps wb-frmvld"></div>
  2. Within the previous <div> element add a <form> element. 
    <form action="#" method="get" id="steps-example"></form>
  3. Within the previous<form> element add your <fieldset> element which will need a <legend> and a <div>. These will become your steps. 
    <fieldset>
  <legend>Client Information</legend>
  <div>(Any content in the div will be hidden.)</div>
</fieldset>

Form elements

Labels

Use labels (a title) to make it clear as to what type of information a user is to enter within an input field.

Correct use

  • Use labels at all times
    • Screen readers have trouble with forms if a label is not included for every input
    • To hide the label, use the .wb-inv class
  • Place the label above the input field. This is the recommended layout
    • This layout allows users to capture both labels and input fields with a single eye movement. While this is the simplest of layouts, users can process the information ten times faster than if the information is left-aligned. This solution uses more vertical space, but does not impact the user experience
  • Shorten label text to the essential minimum
  • Make a statement rather than asking a question
    • For example, use "Child's name" instead of "What is the name of the child?"
  • Use sentence case for example, "First name" not "First Name"
  • Use recognized standard abbreviations, and test abbreviations with non-expert users
  • Place the form objects side-by-side only if there is a need to reduce vertical space
  • Wrap labels and controls in a form group for optimum spacing

Incorrect use

  • Do not use a colon at the end of an input field label
  • Do not use abbreviations, unless absolutely necessary for space reasons
  • Do not abbreviate a critical word. For example, use "Restricted Acct" not "Rstr Account"

Correct use

  • Large checkboxes are the recommended checkbox pattern. In particular, large checkboxes should be used when one or more selections can be made to answer a question
  • Use radio buttons when only one selection can be made in a list
  • Only use tickboxes when there is a single item that can be checked, such as for confirmations
    • Consider using a button instead of tickbox, particularly in cases where it is the primary action or next step on the page

Incorrect use

  • Don't use checkboxes or radio buttons to present options that are not easily comparable. In these cases, use the doormat component
  • Don't use the large checkboxes when building a checklist with items that break across multiple lines or when only a single item can be checked. For these situations, use tickboxes instead.
  • Don't use tickboxes for answers to questions or in checklists. For these situations, use the large checkboxes instead

How to implement

  1. Add class="wb-frmvld" to a div element surrounding the form element 
    <div class="wb-frmvld">
	<form action="#" method="get" id="validation-example">
	...
	</form>
  2. Optional: Wrap each form field name with <span class="field-name">...</span>. This specifies the prefix to use for the error message summary. 
    <label for="fname1">
	<span class="field-name">First Name</span> <strong class="required">(required)</strong>
</label>
  3. Add required="required" to each mandatory form field 
    <input id="fname1" name="fname1" type="text" autocomplete="given-name" required="required" />
